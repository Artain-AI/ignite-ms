# Benchmarking IgniteMS

## Results

### e5-small-v2 (384 dimensions)

p4d.24xlarge (8x A100 40GB), 1M MSMARCO passages, FP16:

| GPUs | IgniteMS | TEI | Speedup |
|-----:|---------:|----:|--------:|
| 1 | 50,127 | 16,648 | 3.0x |
| 8 | 253,578 | 96,492 | 2.6x |

All values are messages/second. These are raw engine throughput: input pre-loaded, no file I/O or output writes. The production section below includes full pipeline overhead.

### GPU scaling (1 vs 8)

| Model | 1 GPU | 8 GPU | Scaling |
|-------|------:|------:|--------:|
| e5-small | 41,842 | 253,950 | 6.1x |
| e5-base | 17,319 | 122,169 | 7.1x |
| e5-large | 5,610 | 41,668 | 7.4x |

Larger models scale closer to linear because GPU inference dominates the pipeline. Smaller models are faster per-GPU but hit the CPU tokenization ceiling at high GPU counts.

## Methodology

### Hardware

- Instance: AWS p4d.24xlarge
- GPUs: 8x NVIDIA A100 40GB, NVLink
- vCPUs: 96
- RAM: 1.1 TB

### Dataset

- Source: MSMARCO v2.1 passages
- Size: 1M passages per run
- Distribution: real English text, variable length (50-256 chars, ~68 tokens average)
- Preprocessing: "passage: " prefix per E5 convention

Real text, not synthetic fixed-length strings. Variable-length input is harder to optimize for because batch utilization depends on length distribution.

### IgniteMS config

- TensorRT 10, FP16
- Model-specific bucket profile generated by `benchmark.py`
- Model-specific batch sizes generated by `benchmark.py`
- CPU-side preprocessing enabled
- No user tuning required for the published benchmark path

### TEI config

- Image: ghcr.io/huggingface/text-embeddings-inference:cuda-1.9
- max-batch-tokens=65536
- max-concurrent-requests=256
- Tuned for throughput on same hardware

Both systems get equal treatment: same hardware, same data, same model, both configured for best throughput.

### Measurement

- Warmup excluded
- Throughput measured at steady state
- Multiple runs, best reported for both
- TEI measured via HTTP client at saturation
- IgniteMS measured from engine telemetry

## How to reproduce

### Docker (automated)

```bash
python3 benchmark.py --model e5-small-v2 --gpu-counts 1,8
```

Downloads everything, runs both engines, reports results. First run: ~10 min (engine compilation). After that, cached.

### Native

```bash
python3 benchmark.py --mode native --model e5-small-v2 --gpu-counts 1,8
```

### Skip TEI

```bash
python3 benchmark.py --skip-tei
```

### Custom data

```bash
python3 benchmark.py --input /path/to/data.jsonl --model e5-small-v2
```

## Caveats

Things that affect results:

- **TensorRT version.** Engine performance depends on TRT version. Docker and native may differ slightly if versions don't match.
- **First run.** Engine compilation takes 3-5 min per bucket/GPU. Cached after that.
- **GPU architecture.** Numbers above are A100. H100 will be faster, T4 will be slower.
- **TEI version.** Performance varies across releases. We pin the version.
- **CPU bottleneck.** At 8 GPUs, CPU-side preprocessing can limit throughput.
- **quickstart.py is not the benchmark.** It's a demo. Use benchmark.py for real numbers.

## What makes it faster than TEI

TEI is a solid embedding server designed for online serving. IgniteMS is a batch engine designed for offline throughput. Different problems, different tradeoffs.

1. **TensorRT inference.** Compiled and fused kernels for the specific GPU architecture and batch shape. Not generic runtime dispatch.

2. **Batching strategy.** We minimize wasted compute per batch. TEI batches dynamically for latency; we batch for throughput.

3. **CPU-GPU co-optimization.** The full pipeline from tokenization through GPU dispatch is tuned as one system. At 8 GPUs the CPU side matters as much as kernel speed.

4. **No network stack.** Reads from disk, writes to disk. No HTTP, no request parsing, no serialization overhead.

5. **Multi-GPU in one process.** One process, N GPUs, shared memory. No per-GPU container orchestration.

6. **Engine caching.** TRT engines compile once and persist across runs. Cached starts are seconds, not minutes.

TEI is an online server you can hit with HTTP requests. IgniteMS is a batch engine you point at data. For batch throughput, the batch tool wins.
